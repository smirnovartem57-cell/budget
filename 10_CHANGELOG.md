## 2026-07-17 — Сопоставимый период в сравнении расходов (ТЗ по графикам)

- «Сравнение периодов» переименовано в «Сравнение с предыдущим периодом» и теперь для
  текущего незавершённого месяца сравнивает **сопоставимые части**: 1–N текущего месяца
  против 1–N предыдущего (день N зажимается по длине предыдущего месяца, если он короче).
  Раньше показывалась только нейтральная пометка «сравниваем неполный период с полным».
- Явная подпись сравниваемых диапазонов: «Сравнение за 1–17 июля и 1–17 июня — одинаковое
  число дней». Разница в ₽ и % считается кодом; вклад категорий («Что повлияло на изменение»)
  берётся по тем же сопоставимым диапазонам. Рост расходов — красный, снижение — зелёный.
- Для завершённых прошлых месяцев и годового вида — прежнее сравнение полных периодов.
- Под «Факт по месяцам» добавлен текстовый вывод: средние расходы по завершённым месяцам и
  месяц-максимум (текущий неполный месяц в среднее не входит, чтобы не искажать).
- Новые чистые хелперы: `sumInMonthUpToDay`, `txInMonthUpToDay`, `daysInMonthKey`,
  `prevMonthKey`, `dayRangeLabel` (детерминированно, исходная точность).

## 2026-07-17 — F7: вход по Face ID / отпечатку (WebAuthn) поверх PIN

- В настройках PIN появился тумблер «Включить вход по Face ID / отпечатку». Он
  показывается только если у устройства есть встроенный аутентификатор

  включён). Успешная проверка снимает блокировку; при любой ошибке/отмене —

- Реализовано на WebAuthn с платформенным аутентификатором; идентификатор ключа
  хранится локально, ничего не уходит на сервер (это локальный замок поверх PIN).
- Гейтинг и вспомогательные функции проверены автоматически; сам биометрический
  запрос доступен только на реальном устройстве с Face ID/Touch ID.

## 2026-07-15 — Долги: частичное погашение (2a) + Покупки: «Куплено» (3a)

  полностью» и «Погашен частично» (ввод суммы через общий лист подтверждения,

- Купленные скрыты из списка, переключатель «Показать купленные (N)». «Вернуть в

## 2026-07-15 — «Получен»: корректировка суммы + правила дублей

### Корректировка суммы при получении
- Кнопка «Получен» на ожидаемом доходе больше не создаёт операцию молча —
  открывается компактное подтверждение (сумма предзаполнена ожидаемой и выделена
  для быстрой правки, дата получения = сегодня, обе редактируемы; input ≥16px,
  тап-зоны ≥44px).
- Операция создаётся с ВВЕДЁННОЙ суммой. Ожидаемый доход хранит обе величины:
  `amount` (ожидаемая, не перезаписывается) и `received_amount` (фактическая) +
  `linked_transaction_id`. Новые колонки добавлены миграцией.
- Если сумма изменена — в списке показывается «Получено X из Y».
- Для ежемесячных доходов правка касается только текущего месяца (строки
  материализованы помесячно, `amount` не трогается → следующий месяц ждёт исходную).

### Правила определения дубля (детерминированно, в копейках, без float)
- Отдельная тестируемая функция `findIncomeCandidates` (+ `moneyEqual`,
  `moneyWithinPct`, `daysApart`).
- Точное совпадение (категория + сумма до копейки + дата ±5 дней) и ровно один
  кандидат → автосвязь без создания операции, тост «Связано с доходом от …» +
  «Отменить» (5 сек).
- Похожее (±3%) либо несколько кандидатов → диалог «Похоже, этот доход уже внесён»
  со списком кандидатов + «Это новый доход».
- Отличие ≥ порога / нет кандидатов → создаётся новая операция.
- Уже связанные операции (`linked_transaction_id`) в поиске кандидатов исключаются;
  поиск ведётся по введённой сумме.

  строке, вторичная информация (дата · комментарий) ниже, действия свёрнуты в одну
